Overview

Problem Statement

Project team members need a way to have contextual conversations about specific tasks. Currently, there is no way to discuss task details, ask questions, or provide updates directly within the task context, forcing teams to use external communication channels where context is easily lost.

Solution

Implement a task comments system that allows team members to add, edit, and delete comments on any task. Comments will support markdown formatting, show timestamps, and be associated with the author's profile.

Target Users

  • Project Managers: Need to ask clarifying questions about task requirements
  • Developers: Need to share implementation notes and technical updates
  • Designers: Need to provide feedback on design tasks
  • QA Engineers: Need to document bugs and testing notes

Success Metrics

  • 80% of tasks have at least one comment within 7 days
  • Average response time to comment questions < 4 hours
  • 90% of users report comments feature is "useful" or "very useful"
  • Reduction in external messaging about task details (measured by Slack mentions)

Technical Considerations

Architecture Decisions

Decision 1: REST API vs GraphQL

Chosen: REST API

Rationale: Existing API is REST-based. Adding GraphQL for one feature increases complexity. REST is sufficient for CRUD operations on comments.

Trade-offs: GraphQL would reduce over-fetching, but adds infrastructure overhead (Apollo Server, schema management).

Alternatives considered: GraphQL (better for complex queries), gRPC (overkill for web clients)

Decision 2: Real-time Updates (WebSockets)

Chosen: Defer to Phase 2

Rationale: Real-time is nice-to-have, not critical. Polling every 30s is acceptable for MVP. WebSockets add complexity (connection management, scaling, fallbacks).

Trade-offs: Users won't see new comments immediately without refresh. Polling increases server load slightly.

Migration path: Add WebSocket server later, use feature flag to enable gradually

Decision 3: Markdown Library

Chosen: marked.js + DOMPurify

Rationale: marked.js is lightweight (20KB), fast, and widely used. DOMPurify sanitizes HTML to prevent XSS.

Trade-offs: Two libraries vs one all-in-one solution. But separation of concerns (parsing vs sanitizing) is cleaner.

Alternatives considered: react-markdown (larger bundle), showdown (older, less maintained)

Decision 4: Soft Delete vs Hard Delete

Chosen: Soft delete (is_deleted flag)

Rationale: Preserves data for auditing. Can restore accidentally deleted comments. Complies with data retention policies.

Trade-offs: Database grows larger (deleted comments still stored). Queries must filter is_deleted=false.

Cleanup strategy: Cron job to hard delete comments >90 days old where is_deleted=true

Database Schema

CREATE TABLE comments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  task_id UUID NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  author_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  content TEXT NOT NULL CHECK (length(content) <= 10000),
  is_deleted BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- Indexes for performance
CREATE INDEX idx_comments_task_id ON comments(task_id) WHERE is_deleted = FALSE;
CREATE INDEX idx_comments_author_id ON comments(author_id);
CREATE INDEX idx_comments_created_at ON comments(created_at DESC);

-- Trigger to update updated_at on edits
CREATE TRIGGER update_comments_updated_at
  BEFORE UPDATE ON comments
  FOR EACH ROW
  EXECUTE FUNCTION update_updated_at_column();

Rollback Plan

If critical issues arise post-deployment:

  1. Feature flag off: Disable comments UI via ENABLE_COMMENTS flag (no code deploy needed)
  2. API remains functional: Existing comments visible but no new additions
  3. Database unchanged: Schema changes are additive (no data loss)
  4. Full rollback: Revert to previous deployment, hide comments section
  5. Gradual re-enable: Fix issue, test, re-enable feature flag for 10% of users, monitor, scale to 100%
